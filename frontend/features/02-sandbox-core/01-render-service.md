# 02-01 Render service

## Цель

Клиентский диспетчер рендера: по `engineId` загружает нужный JS-бандл и выполняет рендер шаблона с JSON-контекстом. Список поддерживаемых движков — хардкод на фронте.

## Зачем

Рендер происходит полностью в браузере — бэкенд не участвует ни в рендере, ни в хранении шаблонного кода во время работы. Каждый движок — отдельный npm-пакет, загружается лениво и кешируется. Добавить новый движок = установить npm-пакет + добавить запись в маппинг — деплой фронтенда.

## Use Cases

- Пользователь набирает шаблон на handlebars — рендер в браузере, без запросов к серверу
- Движок загружается один раз за сессию — повторный рендер без ожидания загрузки
- Шаблон содержит синтаксическую ошибку — сервис возвращает `{ error: string }`, не крашит приложение
- JSON-контекст невалидный — возвращается `{ error: string }` с описанием ошибки парсинга
- Бэкенд добавил новый движок в базу данных — фронт его не поддерживает пока разработчик не добавит бандл

## Критерии готовности

- [ ] `npm install handlebars pug ejs`
- [ ] Рендер работает для handlebars, pug, ejs в браузере
- [ ] Ошибка шаблона или JSON → `{ error: string }`, не исключение
- [ ] Повторный вызов того же движка не перезагружает бандл (кеш)
- [ ] Для каждого engineId определён alias языка Monaco (handlebars → `handlebars`, pug → `pug`, ejs → `html`)
- [ ] Нет обращений к API для получения списка движков

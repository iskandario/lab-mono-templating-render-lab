# 00-05 API-модули

## Цель

Создать API-модули для взаимодействия с бэкендом. Бэкенд хранит только результаты — шаблоны, сохранённые состояния, статистику бенчмарков. Всё вычисление и рендер — на клиенте.

## Зачем

Компоненты и stores не должны знать, откуда приходят данные — из мока или с бэкенда. Переключение через `__USE_MOCK__`. Движки не запрашиваются с бэкенда — они хардкод на фронте (см. `02-01`).

## Модули

- `auth-api` — login, register, logout, forgotPassword, changePassword, getMe
- `templates-api` — CRUD шаблонов пользователя, getPublicTemplates, clone
- `render-runs-api` — сохранение результата бенчмарка, получение истории
- `state-api` — сохранение состояния sandbox на бэкенд, загрузка по id

## Use Cases

- Store вызывает `getMyTemplates()` — в dev получает мок, в prod делает запрос на `/templates`
- После нажатия "Save" пользователем вызывается `saveState()` — результат уходит на бэкенд
- После нажатия "Сохранить запуск" вызывается `saveRun()` — статистика уходит на бэкенд

## Критерии готовности

- [ ] Модули созданы: `auth-api`, `templates-api`, `render-runs-api`, `state-api`
- [ ] `engines-api` отсутствует — движки хардкод на фронте
- [ ] При `__USE_MOCK__ === true` — возвращают мок без сетевых запросов
- [ ] При `__USE_MOCK__ === false` — вызывают http-client с правильными endpoint'ами
- [ ] `npm run type-check` без ошибок
